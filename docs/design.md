## Problem Statement

AI agents pick tools based on description quality, not actual performance. Research shows tools with optimized descriptions get selected 3.6x more often. The entire ecosystem has optimized for discovery (does this tool exist?) but nobody has built selection (which tool is best for this specific task, given these constraints?). Developers currently spend hours manually researching, trying, and comparing tools per category. The decision is made once and never revisited.

## Demand Evidence

Reddit user (senior backend dev, 5 months ago): "Honestly, I feel lost. Without some examples of real-life, day-to-day work, I'll never be able to understand if these skills are going to blend with my way of working." Pattern of similar posts across Reddit, Discord, GitHub where developers express frustration with tool selection. Nobody is asking "what tools exist?" They're asking "which one is right for me?"

## Status Quo

Developers Google it, read blog posts, compare pricing pages manually, try a few tools, and pick one that fits budget. Hours of work per tool category. Decision made once and never revisited even when better options appear or pricing changes. Some ask on Discord/Reddit and get 10 conflicting opinions. Most just use whatever the framework defaults to (LangChain ships with X, so they use X).

## Target User & Narrowest Wedge

**Long-term primary user:** the agent itself. Agents query the recommender to decide which tool to call. Developers integrate once.

**V1 primary user:** developers. They need to trust the system before handing selection to the agent. V1 distribution is developer-facing (dashboard + API). Developers browse, compare, verify the quality data, then integrate the API into their agent's tool-selection logic. Agent-native distribution (MCP) is V2.

This is a deliberate two-phase go-to-market: build developer trust first (dashboard), then earn agent-native distribution (MCP). The demand signal is from developers (Reddit), so the V1 should serve developers. The vision is agent infrastructure.

## Why Skills First

APIs have existed for 15 years. MCPs for over a year. If nobody has built a quality-based API selection layer in all that time, either the market doesn't exist or the direction is wrong. Developers can Google "best stock API" in 30 minutes. The pain isn't big enough.

Skills are different:
- **New and exploding.** Claude Code skills, LangChain tools, CrewAI skills, Cursor rules... the ecosystem is 6 months old and growing fast. Quality is all over the map.
- **Hard to evaluate.** A cold email skill and a code review skill have zero shared quality dimensions. You can't score them the same way. This is the core unsolved problem.
- **Agents can't choose.** An agent faced with 5 cold email skills has no way to know which one produces the best output for its specific use case. It picks based on description. That's broken.

The hard problem, and the moat, is building **category-specific eval rubrics** that let agents choose the right skill automatically. Not one rubric for everything. A rubric per category.

## Definitions

- **Skill:** A prompt-based workflow that an agent executes. Examples: Claude Code skills (SKILL.md files), LangChain tools/chains, CrewAI skills, Cursor rules with attached logic. Skills run LLM calls under the hood and produce structured or unstructured output.
- **Intent:** A user/agent goal that one or more skills can serve. Examples: "ship code", "review PR", "find and fix bugs", "write outreach email". Intents are fuzzy and overlapping, not a rigid taxonomy.
- **Eval Scenario:** A concrete test case: a repo state + task description + expected outcomes. Skills are evaluated by running against scenarios.
- **Elo Rating:** Skills within an intent are ranked via pairwise comparison (A/B eval with LLM judge). Elo captures relative quality without prescribing dimensions upfront.

## Constraints

- **Skills first, APIs/MCPs later.** V1 is entirely about Skills. APIs and MCPs are V2 if validated.
- Eval dimensions are discovered from data (A/B comparisons), not prescribed upfront.
- Config-time recommendation for V1 (quality over speed).
- The eval method + accumulated comparison data is the moat.
- Must support automated eval (agents evaluating skills), not just human eval.
- **Hard requirement:** skill eval runs in sandboxed environment (disposable repo clones, no push access, no real side effects).

## Approaches Considered

### Approach A: MCP-First Wedge
One MCP server exposing "rank_tools" tool. Agents query with task description, get ranked MCP recommendations. MCP-only, no dashboard, no APIs/Skills. Ships in hours. Completeness: 5/10.

### Approach B: API + Developer Dashboard + Agent Interface (CHOSEN)
FastAPI backend with scoring engine. Developer-facing web dashboard for browsing/comparing tools (builds trust). MCP wrapper so agents can query it too. Separate scoring rubrics per tool type (API, Skill, MCP). Two distribution channels: human-browsable and agent-queryable. Completeness: 8/10.

### Approach C: Agent Telemetry Platform
Agents report how tools performed in production. System learns from execution telemetry. Data flywheel. The real moat, but cold start problem and too big for V1. This is the V3 vision. Completeness: 3/10.

### Scoring Dimensions (V1: Skills only)

V1 dimensions are discovered through A/B comparison, not prescribed. However, the eval methods can measure these observable signals:

**Universal (all skills):** completion success, token cost, latency, error rate, sandbox compliance.

**Discovered (emerge from A/B data):** e.g., "edge case coverage" if the judge consistently prefers skills that handle edge cases. These are formalized in Phase 2.

**V2 (future): API and MCP dimensions**
- APIs: accuracy, success rate, latency, reliability, cost per call, rate limits, auth complexity
- MCPs: connection overhead, tool description quality, protocol compliance, startup time

### V1 Scope: Skill Submission Marketplace + Auto-Eval

Users submit their skills. Platform auto-evaluates. Good skills get featured. Agents query the rankings.

**Core loop:** Author submits skill -> auto-eval runs -> score + ranking published -> agents/developers discover it -> more authors submit.

**Cost model:** We only eval what users submit. No crawling, no scraping, no proactive discovery. Eval cost is per-submission, paid by the value of the listing (exposure). At $0.60/eval, this scales.

### Eval Design: Intent-Based, Not Category-Based

The core insight: skills aren't organized into neat categories. gstack has `/ship`, `/review`, `/qa`, `/investigate`. Superpower has its own set. Stoneman has another. They overlap, they differ, they solve the same intent in completely different ways. A rigid taxonomy kills this.

**The model: intent -> candidate skills -> eval -> rank.**

1. **Intent detection.** Agent has a task ("ship this code", "review my PR", "find and fix bugs"). The system maps this to an intent, not a fixed category. Intents are fuzzy, overlapping, and emergent from the skill ecosystem.

2. **Candidate retrieval.** Find all skills that claim to serve this intent. gstack `/ship` and Superpower's ship equivalent and any other shipping skill all become candidates.

3. **Dynamic eval dimensions.** Instead of one rubric per category, the eval dimensions are derived from the intent + the skill's own claims. A shipping skill is evaluated on: does the PR get created? Are tests passing? Is the changelog updated? A review skill: does it catch real bugs? Are suggestions actionable? The dimensions come from what the skill promises to do.

4. **Eval methods (composable):**

| Method | When to use | Example |
|--------|------------|---------|
| **Outcome verification** | Skill produces observable side effects | Did `/ship` actually create a PR? Did tests pass? |
| **LLM judge** | Skill produces text/analysis output | Is the code review insightful? Is the email well-written? |
| **Comparative A/B** | Multiple skills for same intent | Run both on same input, judge which output is better |
| **User signal proxy** | Can observe downstream behavior | Did the user accept the suggestion? Revert it? Edit heavily? |
| **Constraint satisfaction** | Skill has measurable requirements | Completed in <30s? Under 10k tokens? No errors? |
| **Repo fitness** | Skill matches repo characteristics | Does this skill work well with Python monorepos? With small JS projects? |

5. **Eval is contextual.** The same skill can score differently depending on:
   - Repo type (monorepo vs single package, language, framework)
   - Team size (solo dev vs team)
   - Workflow stage (pre-commit vs post-merge vs production)
   - User preferences (thoroughness vs speed)

**This means there's no single leaderboard.** The ranking changes based on context. "Best shipping skill" depends on whether you're a solo dev on a Python project or a team of 10 on a TypeScript monorepo.

**Submission flow:**

1. Author visits the site, submits their skill (GitHub repo URL or SKILL.md upload)
2. Platform validates format, extracts metadata (intent, description, requirements)
3. Auto-eval runs: skill executed against scenarios in sandbox, compared pairwise to existing skills for the same intent
4. Author gets a score card: Elo rating, strengths, weaknesses, judge reasoning
5. If score passes threshold: skill is listed and ranked. If not: author gets feedback on what to improve.

**This is the product.** The submission + auto-eval + feedback loop is what makes authors keep coming back. It's like submitting an app to the App Store, but the review is automated and instant.

**How to bootstrap:**

Phase 1: Seed with 3-5 well-known skill packs (gstack, Superpower, etc.) that we eval ourselves. This creates the initial leaderboard that attracts authors.

Phase 2: Open submissions. Authors submit skills, get ranked against the seed set. Early authors get disproportionate visibility (first-mover advantage incentivizes early submission).

Phase 3: From accumulated A/B data, extract the dimensions that differentiate winners from losers. Formalize into intent-specific eval templates. These templates are the IP.

**Leveraging skill-creator eval:** The existing `/skill-creator` skill has eval infrastructure (variance analysis, A/B testing). Wire the submission pipeline into it so skills created with `/skill-creator` can auto-submit to the marketplace.

### Ranking Algorithm

**Phase 1 (V1): Elo-style pairwise ranking.**

Run comparative A/B evals. For each pair of candidate skills serving the same intent, run both against the same scenario, LLM judge picks the winner. Update Elo ratings. This produces a ranking without predefined dimensions.

Elo parameters: initial rating 1500, K-factor 32, minimum 10 comparisons before a rating is "trusted" (shown with confidence badge). Each comparison is judged 3 times (majority vote) to mitigate LLM judge variance. If all 3 runs disagree (no majority), mark as "inconclusive" and exclude from Elo update.

**Sandbox spec:**
- Each eval runs in a disposable Docker container (or git worktree with restricted permissions)
- No network egress except to LLM APIs (needed for skill execution). All other outbound blocked.
- No push access to any remote. Git operations are local-only.
- Resource limits: 2 CPU, 4GB RAM, 10GB disk, 120s timeout per skill run
- Skills that crash or timeout: scored as "failed" with reason logged. Not scored 0, excluded from that comparison (treated as forfeit).
- Env vars stripped except explicitly whitelisted ones (LLM API keys for execution)

**Cost model (example: "ship code" intent):** 3 skills x 5 scenarios = 15 skill runs. Each pairwise comparison: 3 judge calls. C(3,2) = 3 pairs x 5 scenarios x 3 judge runs = 45 judge calls. Total: ~15 skill runs + 45 judge calls. At $0.01/call avg: ~$0.60 per full intent eval. Affordable for V1.

**LLM judge prompt (pairwise comparison):**
```
You are evaluating two AI skill outputs for the task: "{intent_description}".
Scenario: {scenario_description}

Skill A output: {output_a}
Skill B output: {output_b}

Which output better accomplishes the task? Consider: completeness, correctness,
usefulness, and any observable side effects (files created, commands run, errors).

Respond with EXACTLY one of: "A", "B", or "TIE" followed by a one-sentence reason.
```

**Phase 2: Dimension-weighted scoring.**

Once enough A/B data accumulates, extract the dimensions that consistently differentiate winners from losers. Formalize into intent-specific eval templates with discovered dimensions and learned weights.

**Context-aware filtering:** Before ranking, filter candidates by repo fitness (language, framework, team size). A skill that only works with JavaScript repos shouldn't rank for a Python project.

Fallback: if no skills match the intent, return `no_skills_evaluated: true` with the intent string and suggest the user contribute a skill.

### API Contract

**POST /recommend**
```json
Request:  { "intent": "ship code", "context": { "repo_lang": "python", "team_size": "solo" }, "top": 3 }
Response: { "recommendations": [{ "skill_id": "gstack/ship", "elo": 1520, "eval_summary": { "pr_created": true, "tests_ran": true, ... }, "best_for": "...", "fitness_score": 0.92 }], "intent_detected": "ship_code", "context_applied": true }
```

**POST /compare**
```json
Request:  { "skill_ids": ["gstack/ship", "superpower/deploy"], "scenario": { "repo_lang": "python", "task": "ship bug fix" } }
Response: { "comparison": [{ "skill_id": "...", "elo": ..., "strengths": [...], "weaknesses": [...] }], "winner": "gstack/ship", "judge_reasoning": "..." }
```

**GET /skills?intent=ship_code**
```json
Response: { "skills": [...], "total": 3, "intent": "ship_code", "ranked_by": "elo" }
```

**GET /intents**
```json
Response: { "intents": [{ "name": "ship_code", "skill_count": 3, "eval_count": 25 }, { "name": "code_review", ... }] }
```

**POST /submit** (skill author submits a skill)
```json
Request:  { "repo_url": "https://github.com/user/my-skill", "intent": "ship_code", "description": "..." }
Headers:  { "Authorization": "GitHub OAuth token" }
Response: { "submission_id": "...", "status": "evaluating", "estimated_time_s": 120 }
```

Auth: GitHub OAuth. Author identity = GitHub username. Only the repo owner can submit/update a skill. This prevents impersonation and enables updates (re-submit = re-eval, previous version archived).

**Skill versioning:** Re-submitting the same repo URL triggers a re-eval. New Elo is computed against current field. Old version is archived with its historical Elo. Dashboard shows current version only.

**GET /submit/:id** (check eval status)
```json
Response: { "status": "complete", "elo": 1480, "ranking": 3, "total_in_intent": 5, "scorecard": { "strengths": [...], "weaknesses": [...], "judge_reasoning": "..." }, "listed": true }
```

### Dashboard (V1)

1. **Intent leaderboards** — per-intent Elo rankings with comparison count and confidence badges
2. **Submit skill** — form to submit a skill for evaluation (repo URL + intent + description)
3. **Scorecard view** — after eval, author sees strengths/weaknesses/judge reasoning

The submission flow IS the dashboard. Authors come to submit and see their score. Developers come to browse the leaderboards. Both drive the flywheel.

### Error / Edge Cases

- **No skills for intent:** return `{ "skills": [], "intent": "...", "message": "No skills evaluated for this intent yet. Submit one?" }`
- **Skill crashes during eval:** marked as "failed" for that scenario, excluded from pairwise comparison. Author scorecard shows which scenarios failed and stderr output.
- **All comparisons inconclusive:** (no majority in 3 judge runs) intent is marked "insufficient data", no ranking published until more skills or scenarios are added.
- **Stale data:** each skill record has `eval_date`. Dashboard shows "last evaluated X days ago" badge.

## Open Questions

1. **LLM judge consistency:** 3-run majority vote mitigates variance. Need to measure remaining variance across judge model versions.
2. **Gaming prevention:** Authors could submit skills optimized for the eval scenarios but bad in practice. Mitigation: rotate eval scenarios, include hidden test cases, weight real-world user signals (V2).
3. **Skill execution sandboxing:** Hard requirement. Disposable worktree clones, no push access, timeout limits.
4. **Intent taxonomy growth:** As authors submit skills for new intents, the intent space grows organically. Need a merge/split process for overlapping intents.
5. **Pricing model:** Free to submit and list. Revenue V2: premium placement, verified badges, enterprise API access for agent builders who query at scale.

## Success Criteria

1. **Ranking quality:** For the first evaluated intent, human experts agree with the Elo ranking 80%+ of the time when comparing top-2 skills.
2. **Intent coverage:** 3+ intents with 3+ candidate skills each, evaluated within 8 weeks.
3. **Agent integration proof:** One agent queries the API, selects a skill based on ranking, and produces a measurably better outcome than random selection.
4. **Dimension discovery:** By Phase 2, at least 3 differentiating dimensions per intent cluster are discovered from A/B data (not prescribed).
